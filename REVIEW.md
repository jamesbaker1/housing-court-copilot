# Housing Court Copilot — Consolidated Code & Safety Review

_Reviewed: 2026-06-22. Scope: the now-public GitHub repo + live Cloudflare Workers deploy. Read-only review. Severities below reflect adversarial re-verification of every critical/high finding against the actual source; refuted findings are dropped._

---

## Fixes applied (2026-06-22)

The boundary-hardening pass below was implemented and **independently re-verified** (typecheck + `next build` + `opennextjs-cloudflare build` all green from a fully clean state; local D1 migrations `0001`–`0003` apply cleanly; runtime probes of the schema refine + strip helper; code-read of every changed write path). The four safety invariants now hold **at the API boundary**, not just in the engine.

**Fixed & verified:**

1. **`/api/cases/[id]` is ownership-gated (GET + PATCH + DELETE).** The URL `id` is treated as a loggable locator; every verb requires a per-case capability token (`Authorization: Bearer` / `x-case-token`) or an OTP-verified owner session (`x-owner-session`). Fails **closed** when D1 is bound (uniform `403`, no existence oracle). `lib/auth/session.ts` persists hashes only, constant-time compare. _Verified: code-read of the route + session module; the only open path is the no-D1 dev fallback (`dev_open`, logged), which never occurs on Workers._
2. **Invariant #2 holds at the edge.** PATCH strips `court.court_date_verified` / `court.court_date_source` (and the other safety-owned subtrees) via `stripSafetyOwnedFields`, **and** `CourtSchema` has a cross-field `superRefine`: `court_date_verified===true ⇒ source ∈ {etrack, nyscef}`. _Verified by runtime probe: verified+tenant_entered → REJECT, verified+no-source → REJECT, verified+etrack/nyscef → ACCEPT, unverified+tenant → ACCEPT; strip helper removes the keys._
3. **Invariant #4 holds at the edge.** A client PATCH can no longer write `review.advice_routed` / `advice_detection_log`, `deadlines[].computed_by`, `eligibility.*.determined_by`, or `answer_draft.form_fields[].placed_by` — all stripped before merge and re-validated by `CaseSchema` on persist.
4. **`advice_routed` + the audit event are now persisted server-side in `/api/chat`** (`patchCase({review, audit})` on the authoritative Case). The `review_update` NDJSON frame is now advisory-only; durability/integrity no longer ride on an untrusted client PATCH.
5. **Advice classifier (`screenTurn`) runs before the authoring LLM on `/api/answer` and `/api/defenses`** (Invariant #1), and all copilot output is scanned before send (the normal `proceed` path now buffers→scans→sends one delta instead of token-streaming — a deliberate, necessary tradeoff to scan output).
6. **Rate limiting + Turnstile on the public surface.** `lib/ratelimit.ts` (D1 fixed-window, migration `0003`): per-IP 30/min on intake/chat/defenses/answer/stipulation **and now case-creation** (`POST /api/cases`, added in this pass — it was the one unmetered row/token-flood vector); OTP send capped per-phone 3/hr + per-IP 10/hr + a peek-guarded global 500/day SMS ceiling. `lib/turnstile.ts` siteverify on every public entry point, **fail-closed in production** when `TURNSTILE_SECRET_KEY` is unset.
7. **Court-date persistence bug fixed.** Confirm/correct now passes the **raw ISO** value (`correctedValue ?? rawValue`) to `fieldToCasePatch`, not the rendered long-form display string; `toIsoDate` round-trips, the patch persists with `source="tenant_entered"`, `verified=false`. The client carries the capability token end-to-end.
8. **Retention/deletion posture:** owner-gated `DELETE /api/cases/[id]` (revokes case tokens), Cron retention purge wired in `worker-entry.ts` + `lib/retention.ts` (`[triggers]` daily; never purges `audit.legal_hold`), `[observability]` enabled.
9. **Secret/PII hygiene:** `.dev.vars*` / `.env*` gitignored (root + web); no secrets or owner email in any tracked file (owner email scrubbed from `wrangler.toml`).
10. **Build robustness (fixed in this verification pass):** `worker-entry.ts` imported the gitignored build artifact `./.open-next/worker.js`, which Next's own type-checker resolves strictly — so `next build` (hence `opennextjs-cloudflare build`) failed on any **fresh checkout** or after `rm -rf .open-next` and only passed off stale artifacts. Excluded the Ops Worker entry from the Next build tsconfig and added a dedicated `tsconfig.worker.json` (run by `npm run typecheck`) so it stays type-checked via its ambient declaration. The full pipeline now builds green from a clean state.

**Still open / not addressed here (and operator action required):**

- **Operator must set secrets + redeploy** (not done here, by instruction): `TURNSTILE_SECRET_KEY` + `NEXT_PUBLIC_TURNSTILE_SITE_KEY`, `CF_ACCESS_*`, `CASE_PII_KEY`, `ANTHROPIC_API_KEY`, `TWILIO_*`. **Until Turnstile keys are set in prod, the public API fails closed (`403`)** — that is the intended posture, but it means the app is non-functional for tenants until the keys exist.
- **Operator must apply migrations to REMOTE D1** (local-only verified here): `0001`–`0003` including the new `case_tokens` / `owner_sessions` / `rate_limits` tables. The gate/rate-limit/session logic fails closed without them.
- **Live eTrack / NYSCEF court-date sourcing is still a Tier-2 stub** — no authoritative `verified=true` path exists yet; today every tenant court date is `tenant_entered` / unverified by design. Invariant #2 is enforced, but the *verified* branch is not yet reachable in production.
- **Eligibility engine still unimplemented** (slot empty, no incorrect data produced).
- **Multilingual reach, provider-GET redaction, Idempotency-Key/ETag, and the lower-severity correctness/UX items below remain open** per the tables that follow.

## Overall assessment

**The product's hard safety core is genuinely sound; the soft tissue around it is not.** The four safety invariants that matter most — the model never authors advice, deadlines, eligibility, or a "verified" court date — hold *at the points the invariants are written about*. The advice classifier runs in front of the copilot and fails closed; the model output is suppressed in-band on a hard route; `court_date_verified` is never set true by the model; the deterministic writers in `lib/court-date.ts`, `lib/deadlines.ts`, and `lib/llm/copilot.ts` are correct. Cloudflare Access on `/provider` is verified correctly (JWKS, issuer, audience), and the OTP primitives (hashing, salting, `timingSafeEqual`, TTL, attempt cap, non-enumerable responses) are well-built.

**But the invariants are enforced in the engine, not at the API boundary — and the boundary is wide open.** The single most important finding is that `/api/cases/[id]` is completely unauthenticated for both read and write. That one gap simultaneously (a) exposes full tenant PII to anyone with a case_id, (b) lets an anonymous caller `PATCH` `court.court_date_verified=true` with `source=etrack`, defeating Invariant #2 at the edge because `CourtSchema` has no cross-field refinement, and (c) makes the advice-routing/escalation signal — which the chat route delegates entirely to a fire-and-forget client PATCH — both lossy and forgeable. The deterministic writers are correct, but **they are not the only write path to the persisted Case**, and the alternate path bypasses every invariant.

Beyond that, there is **zero rate limiting or bot protection anywhere** (unbounded Anthropic spend + SMS-bombing once Twilio is live), **no retention/purge/deletion path** for a PII store whose threat model explicitly includes immigration subpoenas, and a **multilingual promise the UI cannot keep** (language is hardcoded `"en"` end-to-end). One safety-critical UX bug is especially nasty: **confirming or correcting the court date silently fails to persist** because the rendered long-form display string fails the `toIsoDate` round-trip — the most important field in the product is the one that doesn't save.

One widely-feared finding was **refuted**: the dual-mode store does *not* silently lose data on Workers. The file fallback hard-fails loudly (EROFS / `notImplemented`) on the first write, so a misconfigured D1 binding produces a 500, not silent data loss. Worth a robustness fix, but not the catastrophe it looked like.

Bottom line: **do not put real tenants in front of this until the boundary is gated and rate-limited.** The legal-safety brain is right; the front door is unlocked.

## Notes on what's solid (don't regress)

- Advice classifier fail-closed + in-band model suppression on `/api/chat`.
- Cloudflare Access JWKS/issuer/audience verification in `lib/auth/access.ts` (prod path correct).

- Deterministic engines (`court-date.ts`, `deadlines.ts`) and the model-never-authors-deadlines/eligibility/advice-line discipline.
- ULID id generation (80 bits crypto random) — not brute-enumerable.
- Immigration data minimized by design (consent-gated, null by default; no SSN/DOB collected).
