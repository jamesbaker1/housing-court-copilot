## Integration Detail — high-priority targets

**HPD Violations / Complaints (wvxf-dwi5, ygpa-z7cr).** Query Socrata SODA JSON server-side, e.g. `https://data.cityofnewyork.us/resource/wvxf-dwi5.json?$where=bbl='{BBL}'`, filtering `CurrentStatus` for open and `Class='C'` for immediately-hazardous; cross-reference complaint dates (`ygpa-z7cr`) against violation dates to assemble a timeline. **Critical caveat (data accuracy):** HPD/DOB open data has known lag, mismatched BBLs, and "closed-but-not-updated" records. An auto-built "violation timeline" presented as court evidence can be *wrong* (e.g., asserting open violations that were actually cured) and could harm a pro se tenant who files it. Every open-data assertion must (a) carry a visible data-accuracy disclaimer, (b) pass through a **tenant "verify before filing" confirmation gate**, and (c) never be auto-asserted into a filing without human review. **Operational catch:** register one free Socrata app token (X-App-Token header, ~1000 req/hr) to avoid HTTP 429; HPD sunset the legacy complaint datasets (`uwyv-629c`/`a2nx-4u46`) ~Jan 31 2024 — use `ygpa-z7cr` and treat dataset IDs as config, not constants.

**HPD Registration + Contacts (tesw-yqqr + feu5-w2e2).** Two-step join: look up BBL in `tesw-yqqr` → get `RegistrationID` → filter `feu5-w2e2` by it for HeadOfficer/IndividualOwner/CorporateOwner/Agent names + business addresses → *registration-defense flag* (no current/valid registration generally bars a nonpayment proceeding). **Catch:** needs the GeoSearch/PLUTO resolver below; surface "registration missing/expired" explicitly but as information to verify, not legal conclusion.

**Address→BBL resolver — GeoSearch + PLUTO/PAD (corrected).** Use the open **GeoSearch** service (`geosearch.planninglabs.nyc`, NYC Planning Labs) for address geocoding and BBL/BIN, joined to **PLUTO/PAD** for parcel attributes. **Do not** rely on the legacy **Geoclient API** — it is deprecated and its app-registration path has been retired; GeoSearch and Geoclient are *not* interchangeable. GeoSearch is open and keyless, which also removes a registration dependency. This resolver is the join key for HPD/DOB/311/ACRIS and a Phase-1 prerequisite.

**Official NY fillable PDF forms + docassemble/AssemblyLine (engine reuse only).** Download the public fillable Answer/OSC/Affidavit PDFs from nycourts.gov, extract AcroForm field names (pdftk/pypdf), build a field-mapping schema. Run on self-hosted `jhpyle/docassemble` Docker (~4GB RAM, ~$10–40/mo) with `docassemble-AssemblyLine` (MIT); use ALWeaver to scaffold and ALDocumentBundle to emit a combined print-ready packet with overflow addendum → PDF/A (also satisfies NYSCEF upload *where the tenant has opted in*). **Catch:** NY's own DIY interviews are proprietary HotDocs/LHI and not forkable — we rebuild equivalent logic; field names change on form revisions, so version-pin and re-validate. **Substantive NY legal review is mandatory and the reviewing attorney must be engaged before this output ships** (RPAPL defenses, deadlines, hard statutory clocks, April-2024 Good Cause).

**Fork MADE — scaffolding only (scope corrected).** MADE encodes **Massachusetts summary-process law** — a different statute, a different Answer form, different discovery rules, no NY Good Cause, no NYC registration defense. The earlier "fork and re-point at NY RPAPL" framing **overstated the head-start**: realistically only the docassemble/ALDocumentBundle *scaffolding and bundling architecture* are reusable; the defense/counterclaim **branching logic must be rebuilt from NY law from scratch**. Treat MADE as a structural reference, not a legal-content starting point.

**JustFix Who Owns What API (api.justfix.org) — endpoints corrected.** The current WoW client (`APIClient.ts`, reviewer-verified 2026-06-22) uses `/api/address`, `/api/address/wowza`, `/api/address/buildinginfo`, and `/api/address/indicatorhistory`. The previously-cited **`/api/address/aggregate?bbl=` endpoint is not in the current client and appears stale/invented — remove it and any build estimate that assumed it.** Geocode address→BBL (via GeoSearch), then call the verified endpoints server-side for portfolio (shell-company names, business addresses, owner names), violations/evictions, and indicator history. Poll `/api/dataset/tracker` for freshness. **Verification status (honest):** the **`/api/dataset/tracker`** endpoint is confirmed live and unauthenticated; the **building-info/portfolio endpoints that actually matter were *not* independently confirmed live in this review** — so "verified live and unauthenticated" should be qualified to the tracker endpoint only until the others are re-checked. The API is undocumented and could add auth/rate-limits without notice — email support@justfix.org before commercial reliance, and scope **NYCDB self-hosting** as the durable fallback (prebuilt NYCDB dumps are CC BY-NC-SA **non-commercial**; commercial use must load from original NYC open-data sources).

**NYC Benefits Screening API + open rules fork (screening only).** Request a free API key (token auth), POST household income/composition/housing, parse returned program codes against the **Benefits & Programs Dataset** to render "how to apply" cards. For data residency/uptime in a courthouse setting, fork `NYCOpportunity/ACCESS-NYC-Rules` (Drools) or the 2025 Python `benefits-screening-api`. **Catch:** eligibility-only — it does **not** submit applications and does **not** fully cover rental vouchers (CityFHEPS/FHEPS/One Shot Deal). Because there is **no submission API anywhere in the ACCESS HRA stack**, every "benefits packet" ends in a **tenant-performed manual upload** to the logged-in ACCESS HRA portal — there is no programmatic pipe. Key-provisioning terms returned 403 to automated fetch; confirm rate limits with NYC Opportunity.

**Language access / translation (new).** The target population is heavily LEP; a city-deployed tool that operates only in English is both a service gap and an equity/compliance exposure (Local Law / language-access expectations for city-facing services). Scope a translation layer in Phase 2: machine translation for UI/intake strings plus a **human-language-access partner** for legally-sensitive document strings and the highest-risk content (machine translation of legal terms is itself a quality risk). Lean on the Benefits & Programs Dataset's multilingual catalog and existing provider language-access resources rather than building from zero.

**SMS reminders (Twilio A2P 10DLC) + court-date sourcing.** Register a Business Profile + A2P Brand + Campaign (nonprofit/government use case), provision a 10DLC/toll-free number, send a 7/3/1-day pre-hearing cadence with plain-language, multilingual copy. **Source dates from two channels, not one:** (1) **eTrack email ingest** (sanctioned) by registering cases and parsing its appearance-reminder emails; and (2) for the **e-filed subset** (post-Feb-2026 L&T e-filing), **NYSCEF case-level public docket data**, which is more authoritative than eTrack for those cases. Do **not** assert a borrowed FTA-reduction magnitude — measure the effect in-context. **Catch:** A2P registration is **mandatory since Feb 1 2025**; reminders are "informational" (lower bar) but still need **prior express consent + STOP handling + consent records**; eTrack delivers email only (no API/webhook) and adding cases is a Cloudflare-protected web form — verify a real housing index number is trackable before relying on it. **Substantive liability note:** a reminder that sends the *wrong date* (mis-parsed, miscalculated against a statutory clock) can directly cause a default — design date-extraction with confirmation and human spot-check, not blind automation. **Do not scrape the live eCourts/WebCivil portal** (Cloudflare + CAPTCHA + ToS bot prohibition).

**LegalServer Online Intake (Trigger API / XML).** Partner per-provider to enable their LegalServer Online Intake site, then use the Online Intake Trigger API (or configured XML import) to launch a pre-filled intake carrying problem codes (LSC CSR + LIST), income/expenses/assets, household, address, and documents (≤100MB). Tag with HO- housing codes. **Catch:** N integrations (one per provider site), not one universal endpoint; Trigger API passes only "limited" fields; full XML import is billable custom work; the deprecated Import API should be avoided. JustFix is already a named LegalServer integration — a potential template/partner. **Note also the funding/eligibility constraint:** if the partner is LSC-funded (only Legal Services NYC in NYC), the program-integrity rule restricts serving undocumented tenants — choose the partner to match the population the integration is meant to serve.

**OCJ/DSS-HRA Brooklyn pilot (deployment venue).** Approach DSS-OCJ and the Tenant Protection Cabinet as a technology pilot partner: pre-court intake/triage + One Shot Deal / voucher / SCRIE-DRIE packet assembly during the court-granted 45-day adjournment, producing a clean handoff file for co-located HRA staff and RTC providers. **Catch:** government procurement is slow; **no application write-API exists**, so this is assemble-and-handoff plus a deep link to ACCESS HRA (tenant uploads), *not* auto-submission — and the "assistance completion/approval" metric is therefore self-report or provider-shared, not tool-captured. Confirm current pilot status/metrics directly (the OCJ FY24/FY25 PDFs returned 403 to automated fetch).

**No programmatic e-filing rail (corrected).** There is **no OASIS LegalXML ECF/EFSP path into NYSCEF** — NYSCEF is a proprietary UCS system without the Tyler/Odyssey-style EFSP vendor ecosystem of other states. The realistic ceiling is the **assisted-upload handoff** for a consenting tenant in an existing case; anything deeper is a bespoke UCS partnership, not a standards integration. Combined with the pro-se exemption from mandatory e-filing (below), the addressable e-filing surface is small — output a print-ready / NYSCEF-uploadable PDF/A and treat assisted upload as an opt-in convenience, not a core funnel.
